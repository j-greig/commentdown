# Commentdown 1.3 release plan

Reconcile the deployed `~/.commentdown/` copy (which drifted ahead of this repo,
uncommitted, dated 2026-06-20) back into the repo as a proper v1.3 release, and
fold in field lessons from two months of daily Claude↔codex use.

## Drift audit (verified 2026-07-10)

Repo (`abbe588`) is at **1.2**. Deployed `~/.commentdown/` is at **1.3** with:

1. **COMMENTDOWN.md**: new `## Order` section — `commentdown.order:
   newest-first | oldest-first`, default newest-first; `## Comments (newest first)`
   heading convention; Agent Start step 1 notes sort order; Append-Only section
   reworded (inserting at declared top is still append-only).
2. **AGENT-LOOP-RUNBOOK.md**: new `## Filing (where the comments live)` section —
   comments live IN the artifact under discussion, never a standalone
   `commentdown.md` or comments-only folder; helper (`cdown.py`) usage note.

## Field lessons (candidate v1.3 material beyond the drift)

- **L1 — the standalone-file mistake keeps recurring.** Agents repeatedly created
  `commentdown.md` / `dev/commentdown/` silos. The Filing rule currently lives
  only in the runbook (optional reading). Candidate: promote to the SPEC —
  a line in `## Entry Locus` ("The host file is the artifact under discussion;
  a dedicated comments-only file is an anti-pattern") + a `## Common Mistakes`
  bullet.
- **L2 — `falsifies:` TODO noise.** The heartbeat helper defaults to
  `falsifies: TODO: name concrete evidence…` when unset, which ships literal
  TODOs into docs (observed today). Candidate: spec guidance that a weak-but-real
  falsifier beats a TODO placeholder, or that helpers must not auto-fill it.
- **L3 — body-echo on routed REQs.** When a REQ's `@actor:` line defaults to the
  title and the body restates the prompt, entries get long. Minor; wording nit
  in Minimal Entry ("route line defaults to title; don't duplicate").

## Plan

2. codex (gpt-5.6-sol) reviews: drift port correctness, whether L1/L2/L3 belong
   in spec vs runbook, adoption-friction check on any new wording.
3. Apply agreed changes, CHANGELOG entry, bump titles/frontmatter to 1.3.
4. Zilla arbitrates anything contested; commit (and push when he says so).

## [REQ] cd-20260710-074653-zilla-discovery-marker · Does CD need a discovery marker for finding all enabled docs later?
@codex: take a position (yay/nay/third way) with exact spec wording if yay.

A month in, how do you find every CD-enabled doc in a repo? Both cases:

YAY (add a marker): `## Comments` is a common heading in ordinary docs, so
grepping it gives false positives; `commentdown:` frontmatter is the obvious
marker but is optional in practice (real example: a doc created today under
this very spec has no frontmatter and is only discoverable via its cd- IDs);
tooling (lint, index, "show open REQs routed to me across the repo") wants a
cheap deterministic scan.

NAY (no new syntax): the format's ethos is plain markdown, zero ceremony; a
required marker is one more thing to forget, and a forgotten marker gives
false negatives, which are worse than grep noise; cd- IDs are already a
high-precision involuntary marker (`grep -rlE '## \[[A-Z]+\] cd-'` hits only
real CD docs, since every entry carries one); frontmatter already exists as
the opt-in marker for docs that want registry/order anyway; spec principle
says new required fields belong in profiles, not core.

Third way to evaluate: no new syntax, add a non-normative "Discovery" note to
the spec documenting the canonical grep (IDs as the marker), and/or make
frontmatter the recommended-but-optional marker with the grep as fallback.

falsifies: a repo-wide scan by either method (frontmatter grep vs entry-ID grep) misses a real CD doc or false-positives on a non-CD doc in practice
tag: discovery-marker

`commentdown.order: newest-first | oldest-first`. Default for documents declaring
Commentdown 1.3: `newest-first`. Documents from earlier versions with no `order`
retain their existing physical order. On upgrade, set `order` to match that
order; never reorder prior entries.

- `newest-first`: insert new entries directly under the comments heading.
- `oldest-first`: append new entries after the last entry.

The comments heading should mirror the setting as `## Comments (newest first)`
or `## Comments (oldest first)`. A bare `## Comments` remains valid in older
documents. `commentdown.order` is authoritative when present.

In either order, append-only means adding entries without rewriting prior
entries.

decorated heading can also evade 1.2 tools that match `## Comments` exactly.
